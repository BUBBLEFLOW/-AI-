解释这段项目
Codex
这个项目叫 Ambisonizer，核心目标是：把输入音频上混成空间音频表示。当前代码主线是 binaural 双耳音频 -> 一阶 Ambisonic B-format 的 W/X/Y/Z 四个通道，之后再按需要解码成立体声或旋转声场。 代码在内层目录：ambisonizer-main - 副本/ambisonizer-main。 项目主线： 1. 数据生成：用 EARS 干净语音和 TAU-SRIR 房间脉冲响应合成训练数据。 2. 训练模型：输入 binaural.wav，目标是 W.wav/X.wav/Y.wav/Z.wav。 3. 推理：模型生成预测的 W/X/Y/Z。 4. 合成：把 W/X/Y/Z 按指定方位角解码成普通立体声。 关键文件： - README.md：项目说明、训练/推理命令、数据格式。 - datasets.py：PyTorch Dataset，读取 binaural.wav 和 W/X/Y/Z.wav，裁剪/补零到固定长度。 - model/seanet.py：核心神经网络 FlowSEANet。 - train.py：训练入口，使用 flow matching 损失。 - batch_inference2_and_synthesize.py：批量推理，并直接生成 W/X/Y/Z 和 stereo 音频。 - synthesize_wxyz_to_stereo.py：单独把 W/X/Y/Z 解码成立体声。 - tau_srir_pipeline/：数据处理流水线，按 01/02/03/04 顺序生成训练集。 模型学习的是：输入 binaural.wav，输出 W.wav, X.wav, Y.wav, Z.wav。当前版本训练的是 conditional flow matching 模型：在双耳音频条件下，从随机噪声逐步生成 W/X/Y/Z。 一句话总结：这是一个空间音频生成项目，用深度学习把双耳音频还原/生成成 Ambisonic 四通道空间声表示，再从这个空间表示渲染到任意扬声器或立体声角度。

现在这个声源是如何转动的 文件：rotating_angles.csv
Codex
这个文件里的旋转规律是： center_azimuth = 0° + 360° * 0.2 * time 也就是每秒转 72°，因为 0.2 Hz = 每 5 秒转一圈。 从你的 rotating_angles.csv 看： 0.0s: center = 0.0°, left = -30.0°, right = 30.0° 1.0s: center = 72.0°, left = 42.0°, right = 102.0° 2.5s: center ≈ -180°，转到背后 4.9s: center = -7.2°，快回到正前方 5.0s: center ≈ 0°，完成一圈 所以它是一个水平面绕圈旋转的效果，仰角 elevation 是 0°，没有上下移动。 更准确地说：脚本不是让模型主动生成一个转圈的声源轨迹，而是用模型生成的 W/X/Y/Z Ambisonic 声场，然后在后处理时不断改变解码角度。 left = center_azimuth - 30° right = center_azimuth + 30° 也就是一个间隔 60° 的虚拟立体声听音方向，围绕声场转一圈。

为啥原来的mrstft基本上是从二开始 现在从十几开始
Codex
主要原因：现在 MRSTFT 的计算对象变了。 原来如果是直接回归，大概率是： MRSTFT(model_output, target) 模型一开始输出可能接近 0 或小幅值信号，所以 MRSTFT 从 2 左右开始并不奇怪。 现在 flow 版本里是： noise = randn_like(target) x_t = (1 - t) * noise + t * target velocity_pred = model(x_t, t, binaural) reconstructed = x_t + (1 - t) * velocity_pred MRSTFT(reconstructed, target) 第一轮模型还没学会时，reconstructed 近似 x_t。而 x_t 里经常含有大量高斯噪声。尤其 t 随机取 0 到 1，当 t 比较小时，x_t 几乎就是噪声。拿噪声和真实音频做 MRSTFT，十几是很正常的。 真实音频 RMS 可能很小，而 torch.randn_like(target) 的标准差是 1.0，所以噪声能量可能比真实音频大很多。

这篇文章四啥思路 文件：zhang25q_interspeech.pdf
Codex
这篇文章是 DiffStereo: End-to-End Mono-to-Stereo Audio Generation with Diffusion Transformer。 核心思路是：用扩散模型直接做 mono-to-stereo，不先生成 Ambisonic/FOA，也不需要方位角、HRTF、文本、图像这些额外条件。 流程： 真实 stereo: L/R ↓ 取平均得到 mono: (L + R) / 2 ↓ STFT 转到复数频谱 ↓ 对 stereo 频谱加高斯噪声 ↓ DiT 扩散 Transformer 学会去噪 ↓ 推理时: mono + 随机噪声 -> 生成 stereo 频谱 -> iSTFT -> stereo audio 它和我们这个项目的最大区别： 我们现在是 binaural -> W/X/Y/Z -> 再解码成立体声/旋转声场。 这篇文章是 mono -> stereo L/R。 它不生成 W/X/Y/Z，也不显式建模三维空间声场。它追求的是端到端把 mono 变成 stereo。 论文结果：DiffStereo 的空间感 MOS 最高，但客观指标不一定最好；Ambisonizer 在一些客观指标和自然度上反而更强。DiffStereo 的主张是：不依赖空间先验和额外标注，只靠 mono 条件，扩散 Transformer 也能生成有明显空间感的 stereo。

现在的模型结构是啥
Codex
现在的默认模型是 FrequencyFlowTransformer，频域条件 Flow Matching Transformer。 结构： 1. 输入 binaural.wav，shape [B, 2, T] 2. 对 binaural 做 STFT，复数谱拆成 real/imag，shape [B, 4, F, N]，作为 condition 3. 对目标 W/X/Y/Z 做 STFT，复数谱拆成 real/imag，shape [B, 8, F, N] 4. 在 WXYZ 频谱上加噪声，得到 noisy spectrum x_t，shape [B, 8, F, N] 5. 把 noisy WXYZ spectrum 和 binaural condition 拼接，shape [B, 12, F, N] 6. 因为 noise_scale = 0.00000002 已经小到几乎等于没有噪声，训练任务被大幅简化了，所以 loss 会下降。 频域 flow 里： noise = noise_scale * randn_like(target_spec) x_t = (1 - t) * noise + t * target_spec velocity_target = target_spec - noise 当 noise_scale 很小时： noise ≈ 0 x_t ≈ t * target_spec velocity_target ≈ target_spec 模型看到的 x_t 里已经包含大量 target 信息，比如 t = 0.8 时 x_t ≈ 0.8 * target_spec。模型只需要把 target 的缩放版本补回 target，难度比“从噪声生成 target”低很多。 所以它下降的原因不是模型突然变强，而是任务难度被 noise scale 降得太低了。

为啥noise scale是0.002损失也下降
Codex
因为 0.002 虽然比 0.00000002 大很多，但在频域 STFT 目标谱的尺度上，它仍然可能是很小的噪声，所以任务还是明显变简单了。 target_spec 经过 log 压缩： compressed_magnitude = stft_log_scale * log1p(magnitude) 默认 stft_log_scale = 0.15。所以 target_spec 的数值范围通常不像 waveform 那样是 0 到 1，它可能大概在 0.001 到 0.3，取决于音频能量和 STFT 参数。 当 noise_scale = 0.002 时，如果很多 target_spec 的有效值比它大，比如 0.02、0.05、0.1，那么噪声并不强。 所以： x_t ≈ t * target_spec + 很小扰动 velocity_target ≈ target_spec 模型还是比较容易学，loss 会下降。 这不一定坏。对你的任务来说，如果最终目标是 binaural -> WXYZ，不追求强随机生成，小噪声 flow 可以作为一种稳定的近似回归式 flow。但要看推理指标和听感。

我的意思是noise_scale如何选择 是不是要靠输入的音频来决定
Codex
不建议让 noise_scale 靠每条输入音频动态决定。 更合理的是：根据训练集里 target_spec 的整体尺度，选一个固定的全局 noise_scale。然后训练和推理都用这个固定值。 原因是： 训练时有 target_spec，可以根据 target_spec 算尺度。 推理时只有 binaural，没有真实 WXYZ target_spec。 如果训练时 noise_scale 依赖 target，推理时就没法一致；如果推理时改成依赖输入 binaural，又会造成训练/推理分布不匹配。 推荐： 1. 在训练集上统计 target_spec 的全局标准差或 RMS。 2. 选择 noise_scale = k * global_target_spec_std。 一般： k = 1.0 难，生成性强 k = 0.3 稳，推荐起点 k = 0.1 更像回归，容易收敛 结论：noise_scale 不应该按每个输入音频临时决定；应该按整个训练集 target_spec 的统计尺度定一个固定值。
